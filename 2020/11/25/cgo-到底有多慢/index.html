cgo 到底有多慢

发表于 2020-11-25 更新于 2020-11-26 分类于 golang

cgo，性能是逃不过的话题，但 cgo 真的很慢么？

Golang 是否能在主代码不变的情况下，通过动态引入包改变部分行为呢？

当我在思考这个问题的时候，我想到了 Java中的 jar 包 和 C 中的动态链接库，他们仅仅需要实现特定的接口，一并加载运行，即可以表现出不同的行为。这时候， cgo 出现在我眼前，允许 Go 程序与 C 的类库交互操作，大大提升 Go 语言的应用场景。cgo 的速度一直被批判，但事实真的是这样么？不服，跑个分。

数据分析

我们可以看到，在单次循环中，GoPlus 一骑绝尘，甩开其他各种 cgo 方法 20-30 倍，cgo 之间区别不是特别大。翻阅相关博客发现，Go 语言调用到 C 代码时必须首先记录 Go 函数入口堆栈的所有细节，然后切换到 C 的堆栈，运行 C 代码，由此带来的开销是昂贵的。

但在百万次的循环中，GoPlus 颓势尽显，被 cgo 超越太多，cgo 之间区别也不是特别大。可以看出随着计算工作的增加，cgo 切换带来的开销显得微不足道，甚至在本次测试中还表现出 C 纯计算的优势。

总结

cgo 不一定真的慢，切换产生的开销是巨大的，但可能被大量的计算所抹平。

这里只是做了速度测试，不代表鼓励大家计算密集的地方都换成 cgo，为了得到这一点儿的性能，你将失去更多的东西，诸如：编辑器不能对虚拟 C 进行编写提示、无法获得跨平台效果、开发人员需要了解更多等。

不同语言适用的场景不同，Go 无法在计算效率上和 C 媲美，但其学习成本、高并发特性是 C 不能企及的。机器永远比人便宜，不能舍本逐末。
